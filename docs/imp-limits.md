Bluesky and the underlying AT Protocol (atproto) enforce distinct tiers of rate limits and data caps. If you are building an aggressive client app that continuously requests video playback, searches, photos, and feeds, you will encounter strict rules across three categories: IP network caps, action points math, and media size quotas.The explicit limits are structured as follows:1. The Hard Network Limit (IP Cap)The Maximum: 3,000 requests per 5 minutes per IP address.How it affects you: This is the ultimate bucket for all raw API traffic (HTTP requests) sent to a standard Personal Data Server (PDS). If your client app refreshes timelines, pulls lookups, and runs profile fetches more than 10 times a second on a single user's connection, the server will return an HTTP 429 ("Too Many Requests") error code.2. The Write & Content Limit (The Points System)When creating content (posting, liking, following, or updating metadata), Bluesky tracks actions using a rolling point-based budget mapped per Decentralized Identifier (DID/Account).The Maximum: 5,000 points per hour and 35,000 points per day per user account.The Math:CREATE an item (a post, photo, or reel/video record) = 3 points.UPDATE an item = 2 points.DELETE an item = 1 point.The Practical Cap: A user can create a maximum of 1,666 total posts, replies, or likes per hour before being throttled.3. Media & Upload Limitations (Videos, Reels, Photos)Bluesky treats "reels" as standard vertical video formats. The network places strict payload limits on media attachments:Video Limits:Daily Ceiling: A hard maximum of 25 video uploads or 10 GB of total video data per day per account, whichever is hit first.Per Post File Size: Max 100 MB per video clip.Duration: Max 3 minutes (180 seconds) per clip.Account Prerequisite: The user's account must have a verified email associated with their PDS before the video endpoint will accept file blobs.Photo Limits:Per Post Quantity: Max 4 images per post.Image File Size: Max 2 MB per image file blob.Note: A single post cannot combine both videos and photos; it must be one or the other.4. Search and Feed Reading LimitsPublic AppView Reads: If you are purely searching for content or fetching public photo/video grids (app.bsky.feed.searchPosts or public timelines), queries hit the AppView (api.bsky.app).The Cap: AppView endpoints have generous, unquantified read limits. As long as your script or client stays under the 3,000 requests/5-minute IP window, it can fetch thousands of search results without penalty.💡 Developer Architecture TipA common bug that triggers sudden 429 rate-limit errors in client apps is session mismanagement.The Trap: Do not call the authorization login endpoint (com.atproto.server.createSession) every time a user refreshes their video feed. Bluesky heavily throttles logins to a maximum of 300 logins per day.The Solution: Authenticate once, save the JSON Web Token (accessJwt and refreshJwt), and utilize the official token refresh process (refreshSession) to sustain the user's session over time
