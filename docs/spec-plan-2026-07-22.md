## 1. EXECUTIVE SUMMARY & PRODUCT VISION

### 1.1 The Core Idea
**VoiceFlow** is a voice-first social media platform that treats short-form audio (1-2 minutes) as the primary content format, not text or images. Users record audio messages that are automatically converted to video files (via browser-based FFmpeg WASM) for compatibility with the AT Protocol's video support, then posted to the network.

### 1.2 Why Voice-First?
- No mainstream social media has made voice the **primary** content format
- Clubhouse proved the desire for audio social, but failed by being ephemeral/live-only
- Twitter Spaces proved audio works in asynchronous contexts
- Instagram voice messages remain an underdeveloped secondary feature
- Voice carries emotional nuance, tone, and personality that text cannot convey
- Accessibility: voice lowers barriers for people who struggle with text

### 1.3 Key Differentiators
1. **Voice-first, not voice-only**: Posts default to voice, but text/image/video are supported
2. **AT Protocol native**: Users own their identity and data through the decentralized protocol
3. **Zero-infrastructure philosophy**: The app operates almost entirely on Bluesky's existing infrastructure
4. **Computed recommendations, not stored**: Matching and discovery happen on-demand via API calls, not databases
5. **Music-powered matching**: Users can express music taste and discover others with similar preferences
6. **Your own brand identity**: Users experience "VoiceFlow," not "a Bluesky client"

### 1.4 Target Audience
- Gen Z and Millennials who prefer audio communication
- Musicians, podcasters, creators who want to share audio snippets
- Communities where voice carries cultural significance
- People who find text-based social media exhausting

## 2. WHY THE AT PROTOCOL?

### 2.1 What We Get For Free (No Backend Required)

| Capability | AT Protocol Provides | Our Responsibility |
|-----------|---------------------|-------------------|
| User accounts & auth | Yes (OAuth + DID) | UI, session management |
| Profiles & handles | Yes | Custom rendering |

| Feed generation | Partially (custom feeds) | Feed algorithm logic |
| Search | Yes (limited) | Enhance with firehose index |

| Moderation | Yes (labels + Ozone) | Client-side filtering |

### 2.2 What We Build Ourselves

| Feature | Approach | Why We Build It |
|---------|----------|-----------------|
| Voice recording & processing | Browser-side WASM | Bluesky doesn't support raw audio |
| Voice-to-video conversion | FFmpeg.wasm | Bluesky supports video, not audio |
| Recommendation algorithms | On-demand computation | No database needed |
| Music taste matching | API-based matching | Computed from public data |
| Custom feed algorithms | Feed generator on free hosting | Unique algorithmic experience |
| Collaborative features | P2P/local-first | Avoid infrastructure costs |
| Unique UI/UX | Custom Next.js app | Product differentiation |

### 2.3 AT Protocol vs. Alternatives

| Aspect | AT Protocol (Bluesky) | ActivityPub (Mastodon) | Nostr |
|--------|----------------------|----------------------|-------|
| **Identity** | DID (portable) | Server-tied (@user@server) | Public-key (cryptographic) |
| **Data portability** | Built-in (PDS migration) | Limited | High (identity = key) |
| **Custom records** | Native (Lexicons) | Difficult | Easy (but untyped) |
| **Developer SDK** | Excellent (@atproto/api) | Moderate | Basic |
| **White-label potential** | Excellent | Moderate | Excellent |
| **Performance** | Good (AppView indexing) | Variable (server load) | Excellent (relays) |
| **Scalability** | Designed for scale | Poor for global feeds | Good |
| **Moderation** | Mature (Ozone, labels) | Server-level | Minimal |

**Verdict:** For a zero-infrastructure, custom-branded social media app that needs to innovate on content types, **AT Protocol is the clear winner**. Its Lexicon system allows us to define voice posts as first-class citizens, and its PDS architecture means users' data is hosted for free on Bluesky's infrastructure.

### 3.2 Data Flow: Creating a Voice Post

```
User records audio
        │
        ▼
MediaRecorder API captures Opus/WebM audio
        │
        ▼
FFmpeg.wasm converts: audio + waveform background → MP4 video
        │
        ▼
Upload MP4 blob to Bluesky PDS (com.atproto.repo.uploadBlob)
        │
        ▼
Create custom record: voiceflow.voice.post
  ├── videoBlob: { ref: CID, mimeType, size }
  ├── transcript: "auto-generated or user-provided"
  ├── duration: 42000 (ms)
  ├── waveformData: [...] (visualization data)
  └── text: optional description
        │
        ▼
Publish to user's repository via BskyAgent
        │
        ▼
Post appears in VoiceFlow feed with rich audio player
Other Bluesky clients see a video post with link to app
```

## 5. AT PROTOCOL DEEP DIVE: WHAT WE GET FOR FREE

### 5.1 Repository Structure

Every user has a **repository** on their Personal Data Server (PDS) that stores all their data as a Merkle Search Tree (MST). The MST is a content-addressed, self-authenticating data structure:

```
Repository (DID:plc:xxxxx)
├── app.bsky.feed.post/
│   ├── 3jvz... (post record)
│   ├── 3jxx... (post record)
│   └── ...
├── app.bsky.feed.like/
│   ├── 3jvy... (like record)
│   └── ...
├── app.bsky.graph.follow/
│   ├── ...
├── app.bsky.graph.block/
│   ├── ...
├── app.bsky.actor.profile/
│   └── self (profile record)
├── voiceflow.voice.post/       ← OUR CUSTOM COLLECTION
│   ├── 3jzz... (voice post)
│   └── ...
└── voiceflow.music.preference/  ← OUR CUSTOM COLLECTION
    └── self (music profile)
```

### 5.2 Key API Endpoints We'll Use Daily

```typescript
// AUTHENTICATION
agent.login({ identifier, password })           // App password login
agent.resumeSession(session)                    // Session restoration
const oauthClient = new NodeOAuthClient({...})  // OAuth (preferred)

// FEEDS & POSTS
agent.getTimeline({ limit, cursor })            // Following feed
agent.getAuthorFeed({ actor, limit })           // User's posts
agent.getPostThread({ uri, depth })             // Threaded view

// SOCIAL GRAPH

agent.getFollowers({ actor, limit, cursor })    // Who follows user

// INTERACTIONS

agent.repost(uri, cid)                          // Repost
agent.app.bsky.feed.getLikes({ uri })           // Who liked

// MEDIA

// SEARCH
agent.app.bsky.actor.searchActors({ term })     // Search users
agent.app.bsky.feed.searchPosts({ q })          // Search posts

// NOTIFICATIONS

// CUSTOM RECORDS

  repo: did,
  collection: 'voiceflow.voice.post',
```

### 5.3 Rate Limits & Constraints

| Endpoint Category | Limit | Source |
|------------------|-------|--------|
| Global API | ~4,000 requests/hour per authenticated session | Empirical |
| `getTimeline` | 100 per call, cursor for pagination | API spec |
| `uploadBlob` | ~1MB for images, ~100MB for video | PDS config |
| `searchPosts` | Max 25 results per call | API spec |
| `getFollowers` | 100 per page, max depth unlimited | API spec |
| Session duration | Token expires ~2 hours (refresh available) | OAuth spec |

**Strategy:** Use `@tanstack/react-query` for aggressive caching, deduplication, and stale-while-revalidate to minimize API calls. Cache feed data in IndexedDB with 5-minute TTLs.

### 5.4 The Firehose & Jetstream

**Jetstream** is the recommended way to consume real-time AT Protocol data:

```typescript
// Connecting to Jetstream (simplified firehose)
import { Jetstream } from '@skyware/jetstream';

const jetstream = new Jetstream({
  ws: 'wss://jetstream1.us-east.bsky.network/subscribe',
  // Filter to only our custom collections + essential Bluesky ones
  wantedCollections: [
    'voiceflow.voice.post',
    'app.bsky.feed.post',
    'app.bsky.feed.like',
    'app.bsky.graph.follow',
  ],
});

jetstream.on('commit', (event) => {
  switch (event.commit.collection) {
    case 'voiceflow.voice.post':
      // A new voice post was created anywhere on the network
      break;
    case 'app.bsky.feed.like':
      // Someone liked a post (check if it's our user's post)
      break;
  }
});

jetstream.start();
```

**Where to run Jetstream:** On a dedicated background process (Railway free tier, Fly.io free tier, or a cloud VM), NOT inside Next.js serverless functions (timeout limits).

**Alternative:** For initial launch, simply **poll** the notifications API every 30-60 seconds. Jetstream only becomes necessary when scaling to thousands of concurrent users.

### 6.4 Registering Custom Lexicons

Custom Lexicons **do NOT need to be registered with a central authority**. You simply:
1. Define the Lexicon JSON files in your project (`lexicons/voiceflow/voice/post.json`)
2. Use the `@atproto/lex` CLI to generate TypeScript types: `lex gen-types`
3. Create records using the collection name (`voiceflow.voice.post`)
4. Other apps that understand your Lexicon will render them; others will ignore them

This is the **power** of the AT Protocol — applications can coexist on the same infrastructure without interfering.

## 7. AUTHENTICATION SYSTEM

### 7.1 OAuth Flow (Recommended)

The AT Protocol now uses OAuth as the primary authentication mechanism (app passwords are being deprecated):

```
User enters their Bluesky handle (@user.bsky.social)
        │
        ▼
Next.js server resolves handle → DID → PDS URL
        │
        ▼
Server redirects user to their PDS's OAuth authorization page
        │
        ▼
User approves on their PDS
        │
        ▼
PDS redirects to our callback URL
        │
        ▼
Server exchanges authorization code for tokens
        │
        ▼
Store session in httpOnly cookie (DID + refresh token)
        │
        ▼
User is logged in to VoiceFlow
```

### 7.2 Next.js Implementation

```typescript
// lib/auth/client.ts
import { NodeOAuthClient } from '@atproto/oauth-client-node';

export const oauthClient = new NodeOAuthClient({
  clientMetadata: {
    client_name: 'VoiceFlow',
    client_id: `https://voiceflow.app/oauth-client-metadata.json`,
    client_uri: 'https://voiceflow.app',
    redirect_uris: ['https://voiceflow.app/oauth/callback'],
    grant_types: ['authorization_code', 'refresh_token'],
    response_types: ['code'],
    token_endpoint_auth_method: 'none',
    application_type: 'web',
    dpop_bound_access_tokens: true,
  },
  keyset: await loadKeySet(), // Load signing keys
  sessionStore: redisSessionStore, // Or SQLite for dev
});

// app/oauth/login/route.ts
export async function GET(request: Request) {
  const url = new URL(request.url);
  const handle = url.searchParams.get('handle');
  
  if (!handle) {
    return Response.redirect(new URL('/login', request.url));
  }
  
  const { url: authUrl } = await oauthClient.initiate(handle, {
    scope: 'atproto transition:generic',
  });
  
  return Response.redirect(authUrl);
}
```

### 7.3 Session Management Strategy

- **Server-side**: Store DID and refresh token in encrypted httpOnly cookies
- **Client-side**: Read DID from a non-sensitive cookie to detect auth state
- **Middleware**: Check cookie in Next.js middleware for protected routes
- **Token refresh**: Implement automatic refresh when agent calls return 401
- **Logout**: Clear cookies and optionally revoke OAuth tokens

### 7.4 Fallback: App Passwords (During Development)

For early development before OAuth is fully implemented:

```typescript

  identifier: 'user.handle',

// Store session in httpOnly cookie

```

```typescript
export async function convertAudioToVideo(
  audioBlob: Blob,
  options?: {
    background?: string; // gradient or image
    showWaveform?: boolean;
    avatarUrl?: string;
  }
): Promise<Blob> {
  const ffmpeg = await loadFFmpeg();
  
  // Write audio file to FFmpeg's virtual filesystem
  await ffmpeg.writeFile('input.webm', await fetchFile(audioBlob));
  
  // Generate a background image (gradient based on user's color)
  // In production, use a canvas-generated PNG
  await ffmpeg.writeFile('bg.png', await generateBackgroundImage(options?.background));
  
  // Run FFmpeg: combine background image + audio → MP4 video
  // -loop 1: loop the image across the entire video duration
  // -shortest: stop when audio ends
  // -pix_fmt yuv420p: ensure compatibility across all devices
  const ffmpegArgs = [
    '-loop', '1',
    '-i', 'bg.png',
    '-i', 'input.webm',
    '-c:v', 'libx264',
    '-preset', 'ultrafast', // Prioritize speed over compression in browser
    '-tune', 'stillimage',
    '-c:a', 'aac',
    '-b:a', '128k',
    '-pix_fmt', 'yuv420p',
    '-shortest',
    'output.mp4',
  ];
  
  // If waveform visualization is requested
  if (options?.showWaveform) {
    // More complex filter for waveform overlay
    ffmpegArgs.splice(-2, 0,
      '-filter_complex',
      '[0:v]scale=720:1280,setsar=1[bg];' +
      '[1:a]showwaves=s=720x200:mode=cline:rate=25:colors=#6C63FF|#FF6B6B[waves];' +
      '[bg][waves]overlay=0:1080[out]',
      '-map', '[out]',
      '-map', '1:a',
    );
  }
  
  await ffmpeg.exec(ffmpegArgs);
  
  // Read the output file
  const data = await ffmpeg.readFile('output.mp4');
  
  // Clean up virtual filesystem
  await ffmpeg.deleteFile('input.webm');
  await ffmpeg.deleteFile('bg.png');
  await ffmpeg.deleteFile('output.mp4');
  
  return new Blob([data], { type: 'video/mp4' });
}
```

### 11.1 Content-Based Recommendation (Zero Storage)

```typescript
// algorithms/recommendation.ts

interface UserContentProfile {
  did: string;
  hashtags: Map<string, number>;
  bigrams: Map<string, number>;
  totalPosts: number;
}

export class RecommendationEngine {
  // Build a content profile from recent posts (no storage)
  private async buildProfile(agent: BskyAgent, did: string): Promise<UserContentProfile> {
    const feed = await agent.getAuthorFeed({ actor: did, limit: 50 });
    
    const hashtags = new Map<string, number>();
    const bigrams = new Map<string, number>();
    
    for (const item of feed.data.feed) {
      const text = item.post.record.text || '';
      const postHashtags = extractHashtags(text);
      
      for (const tag of postHashtags) {
        hashtags.set(tag, (hashtags.get(tag) || 0) + 1);
      }
      
      // Extract word bigrams for content similarity
      const words = text.toLowerCase().split(/\s+/).filter(w => w.length > 3);
      for (let i = 0; i < words.length - 1; i++) {
        const bigram = `${words[i]} ${words[i + 1]}`;
        bigrams.set(bigram, (bigrams.get(bigram) || 0) + 1);
      }
    }
    
    return { did, hashtags, bigrams, totalPosts: feed.data.feed.length };
  }
  
  // Compute cosine similarity between two profiles
  private cosineSimilarity(a: UserContentProfile, b: UserContentProfile): number {
    // Create union of all features
    const allFeatures = new Set([
      ...a.hashtags.keys(),
      ...b.hashtags.keys(),
      ...a.bigrams.keys(),
      ...b.bigrams.keys(),
    ]);
    
    let dotProduct = 0;
    let magA = 0;
    let magB = 0;
    
    for (const feature of allFeatures) {
      const valA = (a.hashtags.get(feature) || 0) * 3 + (a.bigrams.get(feature) || 0);
      const valB = (b.hashtags.get(feature) || 0) * 3 + (b.bigrams.get(feature) || 0);
      
      dotProduct += valA * valB;
      magA += valA * valA;
      magB += valB * valB;
    }
    
    if (magA === 0 || magB === 0) return 0;
    return dotProduct / (Math.sqrt(magA) * Math.sqrt(magB));
  }
  
  // Get recommendations (on-demand, no storage)
  async getRecommendations(agent: BskyAgent): Promise<string[]> {
    const myDid = agent.session!.did;
    const myProfile = await this.buildProfile(agent, myDid);
    
    // Get users I follow
    const followsResponse = await agent.getFollows({ actor: myDid, limit: 100 });
    const followedUsers = followsResponse.data.follows.map(f => f.did);
    
    // Build profiles for a subset of followed users (to avoid rate limits)
    const followedProfiles = await Promise.all(
      followedUsers.slice(0, 30).map(did =>
        this.buildProfile(agent, did).catch(() => null)
      )
    );
    
    // Find which followed users have the most similar content to me
    const similarities: Array<{ did: string; score: number }> = [];
    for (const profile of followedProfiles) {
      if (profile) {
        similarities.push({
          did: profile.did,
          score: this.cosineSimilarity(myProfile, profile),
        });
      }
    }
    
    // Find users that my similar follows follow (but I don't)
    const topSimilarFollows = similarities
      .sort((a, b) => b.score - a.score)
      .slice(0, 5)
      .map(s => s.did);
    
    const suggestions = await this.getAgent.suggestionsFromUsers(
      agent, topSimilarFollows, followedUsers
    );
    
    return suggestions;
  }
  
  private async suggestionsFromUsers(
    agent: BskyAgent,
    sourceUsers: string[],
    excludeUsers: string[]
  ): Promise<string[]> {
    const excludeSet = new Set([agent.session!.did, ...excludeUsers]);
    const candidates = new Map<string, number>();
    
    for (const did of sourceUsers) {
      const follows = await agent.getFollows({ actor: did, limit: 50 });
      for (const follow of follows.data.follows) {
        if (!excludeSet.has(follow.did)) {
          candidates.set(follow.did, (candidates.get(follow.did) || 0) + 1);
        }
      }
    }
    
    return [...candidates.entries()]
      .sort((a, b) => b[1] - a[1])
      .slice(0, 20)
      .map(([did]) => did);
  }
}
```

```typescript
// algorithms/voicePattern.ts
// Detect voice characteristics for matching users by audio style

export function analyzeVoiceFeatures(audioBuffer: AudioBuffer) {
  const channelData = audioBuffer.getChannelData(0);
  const sampleRate = audioBuffer.sampleRate;
  
  // Compute RMS energy (volume)
  let sumSquares = 0;
  for (let i = 0; i < channelData.length; i++) {
    sumSquares += channelData[i] * channelData[i];
  }
  const rms = Math.sqrt(sumSquares / channelData.length);
  
  // Compute spectral centroid (brightness of voice)
  const fftSize = 2048;
  const numFrames = Math.floor(channelData.length / fftSize);
  let spectralCentroid = 0;
  
  for (let frame = 0; frame < numFrames; frame++) {
    const start = frame * fftSize;
    const frameData = channelData.slice(start, start + fftSize);
    // Simple spectral analysis
    let weightedSum = 0;
    let magnitudeSum = 0;
    for (let i = 0; i < fftSize / 2; i++) {
      const magnitude = Math.abs(frameData[i]);
      weightedSum += i * magnitude;
      magnitudeSum += magnitude;
    }
    spectralCentroid += (weightedSum / magnitudeSum) / numFrames;
  }
  
  return {
    averageVolume: rms,
    spectralCentroid: (spectralCentroid / sampleRate) * fftSize,
    duration: audioBuffer.duration,
    zeroCrossingRate: computeZeroCrossingRate(channelData),
  };
}
```

## 12. MUSIC TASTE MATCHING SYSTEM

### 12.1 Architecture

```
User sets favorite songs/genres in their VoiceFlow profile
        │
        ▼
Stored as voiceflow.music.preference custom record in their PDS
        │
        ▼
When a user wants to find music matches:
        │
        ▼
Fetch their own music preferences
        │
        ▼
Fetch music preferences from potential matches
  (followed users, users in same hashtag communities)
        │
        ▼
Compute compatibility score:
  - Genre overlap (weighted heavily)
  - Artist overlap (weighted heavily)
  - Track overlap (weighted most heavily)
        │
        ▼
Return ranked list of music-similar users
```

### 12.2 Music Matching Implementation

```typescript
// services/musicMatch.ts

interface MusicProfile {
  did: string;
  favorites: Array<{
    title: string;
    artist: string;
    platformUrl: string;
  }>;
  genres: string[];
}

export class MusicMatchService {
  constructor(private agent: BskyAgent) {}
  
  // Fetch a user's music preference record
  async getMusicProfile(did: string): Promise<MusicProfile | null> {
    try {
      const response = await this.agent.com.atproto.repo.getRecord({
        repo: did,
        collection: 'voiceflow.music.preference',
        rkey: 'self',
      });
      
      return response.data.value as MusicProfile;
    } catch {
      return null; // User hasn't set up music profile
    }
  }
  
  // Compute music compatibility between two users
  computeMusicCompatibility(a: MusicProfile, b: MusicProfile): number {
    let score = 0;
    let maxScore = 0;
    
    // Genre overlap (weight: 30%)
    const aGenres = new Set(a.genres.map(g => g.toLowerCase()));
    const bGenres = new Set(b.genres.map(g => g.toLowerCase()));
    const genreOverlap = [...aGenres].filter(g => bGenres.has(g)).length;
    const maxGenres = Math.max(aGenres.size, bGenres.size, 1);
    score += (genreOverlap / maxGenres) * 30;
    maxScore += 30;
    
    // Artist overlap (weight: 40%)
    const aArtists = new Set(a.favorites.map(f => f.artist.toLowerCase()));
    const bArtists = new Set(b.favorites.map(f => f.artist.toLowerCase()));
    const artistOverlap = [...aArtists].filter(a => bArtists.has(a)).length;
    const maxArtists = Math.max(aArtists.size, bArtists.size, 1);
    score += (artistOverlap / maxArtists) * 40;
    maxScore += 40;
    
    // Track overlap (weight: 30%)
    const aTracks = new Set(a.favorites.map(f => `${f.artist}:${f.title}`.toLowerCase()));
    const bTracks = new Set(b.favorites.map(f => `${f.artist}:${f.title}`.toLowerCase()));
    const trackOverlap = [...aTracks].filter(t => bTracks.has(t)).length;
    const maxTracks = Math.max(aTracks.size, bTracks.size, 1);
    score += (trackOverlap / maxTracks) * 30;
    maxScore += 30;
    
    return maxScore > 0 ? (score / maxScore) * 100 : 0;
  }
  
  // Find music matches from followed users
  async findMusicMatches(limit = 20): Promise<Array<{ did: string; compatibility: number; profile: MusicProfile }>> {
    const myDid = this.agent.session!.did;
    const myProfile = await this.getMusicProfile(myDid);
    if (!myProfile) return []; // User hasn't set up music preferences
    
    // Get users I follow
    const followsResponse = await this.agent.getFollows({ actor: myDid, limit: 100 });
    const followDids = followsResponse.data.follows.map(f => f.did);
    
    // Fetch music profiles for followed users
    const profiles = await Promise.all(
      followDids.map(did => this.getMusicProfile(did))
    );
    
    // Compute compatibility and rank
    const matches: Array<{ did: string; compatibility: number; profile: MusicProfile }> = [];
    for (let i = 0; i < profiles.length; i++) {
      const profile = profiles[i];
      if (profile) {
        matches.push({
          did: followDids[i],
          compatibility: this.computeMusicCompatibility(myProfile, profile),
          profile,
        });
      }
    }
    
    return matches.sort((a, b) => b.compatibility - a.compatibility).slice(0, limit);
  }
}
```

```typescript
// services/realtime/JetstreamService.ts (runs on a free Node.js server)
import { Jetstream } from '@skyware/jetstream';

export class RealtimeService {
  private jetstream: Jetstream;
  private subscribers: Map<string, Set<(event: any) => void>> = new Map();
  
  constructor() {
    this.jetstream = new Jetstream({
      ws: 'wss://jetstream1.us-east.bsky.network/subscribe',
      wantedCollections: [
        'app.bsky.feed.like',
        'app.bsky.feed.post',
        'app.bsky.graph.follow',
        'app.bsky.feed.repost',
        'voiceflow.voice.post',
      ],
    });
    
    this.setupHandlers();
  }
  
  private setupHandlers() {
    this.jetstream.on('commit', (event) => {
      const { commit } = event;
      
      // Notify relevant subscribers
      const subscribers = this.subscribers.get(commit.collection);
      if (subscribers) {
        for (const callback of subscribers) {
          callback(event);
        }
      }
    });
  }
  
  subscribe(collection: string, callback: (event: any) => void) {
    if (!this.subscribers.has(collection)) {
      this.subscribers.set(collection, new Set());
    }
    this.subscribers.get(collection)!.add(callback);
    
    return () => this.subscribers.get(collection)?.delete(callback); // Unsubscribe
  }
  
  start() {
    this.jetstream.start();
  }
  
  stop() {
    this.jetstream.stop();
  }
}
```

```typescript
// Client-side backup scheduler (runs in background)
export function scheduleBackup(agent: BskyAgent, intervalHours = 24) {
  const backupKey = `backup_${agent.session!.did}`;
  
  const runBackup = async () => {
    const lastBackup = localStorage.getItem(backupKey);
    const hoursSinceBackup = lastBackup
      ? (Date.now() - parseInt(lastBackup)) / (1000 * 60 * 60)
      : Infinity;
    
    if (hoursSinceBackup >= intervalHours) {
      try {
        const repo = await exportUserRepository(agent);
        
        // Store in IndexedDB as a fallback
        const backupDb = await openBackupDatabase();
        await backupDb.put('backups', {
          did: agent.session!.did,
          data: await repo.arrayBuffer(),
          timestamp: Date.now(),
        });
        
        localStorage.setItem(backupKey, Date.now().toString());
        console.log('Backup completed successfully');
      } catch (error) {
        console.error('Backup failed:', error);
      }
    }
  };
  
  // Run backup check on app startup
  runBackup();
  
  // Schedule periodic checks
  return setInterval(runBackup, 60 * 60 * 1000); // Check every hour
}
```

## CONCLUSION

VoiceFlow is not just technically feasible — it's architecturally elegant. By leveraging the AT Protocol as our infrastructure layer, we can build a voice-first social media platform with **zero server costs** for the initial launch phase. The key insight is that the AT Protocol provides identity, storage, social graph, and data distribution for free — we only need to build the unique product layer.

The most important strategic decisions are:

1. **Define custom Lexicons early** — `voiceflow.voice.post`, `voiceflow.music.preference`, and `voiceflow.actor.bio` are our data foundation. Get these right.

2. **Implement OAuth from day one** — Don't rely on app passwords. The AT Protocol is migrating away from them.

3. **Voice processing in the browser** — FFmpeg.wasm converts audio to video client-side, avoiding any server cost. This is our secret weapon.

4. **Computation over storage** — Recommendations, matching, and discovery happen on-demand via API calls, not SQL queries. This eliminates the need for a database.

5. **Brand your own experience** — Users should never know they're on the AT Protocol. VoiceFlow is its own platform.
